# 命名备选清单: Cross-Review 产品评审草案

> 状态：评审草案  
> 目的：先把命名空间说清楚，后续再由用户拍板，不在本文件内直接定名。

## 命名层级

当前方案把命名拆成 4 层，避免一个名字同时承担所有职责：

1. **品牌名 / 产品名**
   - 面向 repo、README 首页、发布页、长期识别

2. **能力名**
   - 面向用户理解产品在做什么
   - 当前能力语义候选集中在 `cross-review`

3. **内核语义名**
   - 面向架构和实现
   - 当前语义候选集中在 `verification loop`

4. **Sopify 集成名**
   - 面向 adapter、profile、集成包命名
   - 不要求和独立产品名完全一致

## 评审标准

本轮命名评审使用以下标准：

1. **宿主中立性**
   - 是否会让人误以为它只能服务 `Sopify`

2. **语义清晰度**
   - 是否能让用户快速理解产品在做什么

3. **品牌可记忆性**
   - 是否像一个独立产品，而不是内部模块名

4. **扩展性**
   - 是否能覆盖 `design-review / code-review / final-audit`

5. **工程适配性**
   - 是否适合作为 repo 名、包名、模块名、文档标题

## 命名路线

### 路线 A：方法型命名

强调“怎么审”，而不是“审什么”。

特点：

- 优势：更能体现产品差异化方法
- 风险：如果做主品牌，有时会略泛、略像能力名

代表：

- `cross-review`
- `crosscheck`
- `independent-review`

### 路线 B：对象型命名

强调“审的对象是工作产物 / artifact”。

特点：

- 优势：范围边界清晰，不只局限于代码
- 风险：更像工程术语，品牌感偏弱

代表：

- `artifact-review`
- `artifact-audit`
- `workproduct-review`

### 路线 C：品牌型命名

强调产品记忆点，通过副标题解释能力。

特点：

- 优势：适合做独立品牌，宿主中立
- 风险：第一眼解释成本更高，需要副标题支撑

代表：

- `TwinLens`
- `SecondSight`
- `Verifact`
- `Veriview`

## Shortlist

以下 shortlist 不是最终推荐顺序，而是当前最值得保留到后续决策的候选。

| 候选 | 类型 | 核心含义 | 优点 | 风险 | 更适合作为 |
|---|---|---|---|---|---|
| `cross-review` | 方法型 | 用独立、交叉的方式复核已有产物 | 直观，最贴当前产品方法论，和现有方案延续性最好 | 作为主品牌略泛，品牌独特性一般 | 能力名 / 命令名 / 产品副标题 |
| `artifact-review` | 对象型 | 审查 AI 工作产物，而不仅是代码 | 范围准，能自然覆盖 plan/design/task/audit | 偏冷、偏工程术语，品牌感弱 | 内部设计语言 / 核心产品候选 / 文档标题 |
| `TwinLens` | 品牌型 | 第二视角、双镜头复核 | 品牌感强，贴“第二视角”心智，不被单一工件绑死 | 第一眼不如 `cross-review` 直白 | 品牌名 / repo 名 / 产品首页名 |
| `SecondSight` | 品牌型 | 第二眼、再看一遍的能力 | 品牌气质强，贴“独立复核” | 稍偏抽象，需要副标题解释 | 品牌名 / 营销名 |
| `Verifact` | 品牌型 | `verify + artifact` 的组合词 | 兼顾工程语义与品牌感，和 artifact 边界贴合 | 造词感更强，首轮理解成本略高 | 品牌名 / repo 名 |

## 候选逐项说明

### 1. `cross-review`

**意思**

- 它强调产品的核心方法：不是作者原地自审，而是通过第二视角、独立上下文、交叉验证来复核产物。

**优点**

- 用户一眼能理解“这是一种 review 能力”
- 与当前方案、思路、对话上下文完全一致
- 很适合作为能力名与命令名

**风险**

- 作为主品牌稍泛
- 更像“能力描述”，不完全像“独立产品品牌”

**当前最适合的位置**

- 能力名
- 副标题
- 命令名

### 2. `artifact-review`

**意思**

- 它强调产品审查的是工作产物，不只是代码，也包括 plan、design、task result、final audit scope。

**优点**

- 语义范围和当前产品边界高度一致
- 宿主中立
- 能很好支撑“不是只给 Sopify 用”的方向

**风险**

- 较像工程模块名
- 对非工程型用户不如 `cross-review` 直观

**当前最适合的位置**

- 内部设计语言
- 核心产品候选
- 文档章节 / repo 候选

### 3. `TwinLens`

**意思**

- 用“双镜头 / 第二镜头”表达第二视角复核。

**优点**

- 品牌感强
- 很贴“第二视角”和“独立验证”的心智
- 不被 code / Sopify / workflow 绑死

**风险**

- 首次看到不知具体做什么
- 必须配副标题

**当前最适合的位置**

- 品牌名
- repo 名
- 独立产品首页名

建议副标题示例：

`TwinLens — Cross-review engine for AI work products`

### 4. `SecondSight`

**意思**

- 用“第二眼 / 再看一遍”的感觉表达独立复核。

**优点**

- 品牌气质更成熟
- 强调“看见生产者没看见的东西”

**风险**

- 偏抽象
- 与 review / artifact 的直接关联弱于 `TwinLens`

**当前最适合的位置**

- 品牌名
- 宣传层名称

### 5. `Verifact`

**意思**

- `verify + artifact`

**优点**

- 同时保留验证和工件两个核心概念
- 比 `artifact-review` 更有品牌感
- 比纯创意词更贴工程语义

**风险**

- 造词属性更强
- 第一轮需要解释

**当前最适合的位置**

- 品牌名
- repo 名

## 当前推荐理解

本轮不做最终选择，但从“命名层级拆分”角度，当前最合理的理解是：

### 方案 1：稳妥型

- 品牌名：`CrossReview`
- 能力名：`cross-review`
- 内核语义：`verification loop`

适合：先求清晰，再考虑品牌升级。

### 方案 2：工程中立型

- 品牌名：`ArtifactReview`
- 能力名：`cross-review`
- 内核语义：`verification loop`

适合：更强调对象边界和宿主中立性。

### 方案 3：品牌化型

- 品牌名：`TwinLens`
- 副标题：`Cross-review engine for AI work products`
- 内核语义：`verification loop`

适合：从一开始就把它当独立产品品牌来做。

## 暂不推荐

以下名字当前不建议作为主品牌：

- `sopify-cross-review`
  - 风险：若最终不是只给 Sopify 用，宿主绑定感过强

- `verification-loop`
  - 风险：更像内部机制名，不像产品品牌

- `artifact-audit`
  - 风险：过于偏后期审计，不够覆盖 design / task review

- `workproduct-review`
  - 风险：过长，表达虽准但品牌感较弱

## 后续待决策问题

以下问题留待后续统一拍板：

1. 主品牌是否继续保留 `sopify-` 前缀
2. 核心产品名是否走：
   - 方法型
   - 对象型
   - 品牌型
3. `cross-review` 是否固定只作为能力名，而不再兼任主品牌
4. 是否采用“品牌名 + 副标题”的双层命名方式

在这些问题未决前，本文件中的 shortlist 只作为评审输入，不视为最终定名。
