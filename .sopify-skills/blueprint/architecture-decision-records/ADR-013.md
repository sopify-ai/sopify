# ADR-013: Product Positioning — Evidence & Authorization Layer

状态: 已确认（2026-05 定位从 "Workflow Control Plane" 精确化为 "Evidence & Authorization Layer"）
日期: 2026-04（初始），2026-05（定位精确化）

## 背景

Sopify 需要明确自身在 AI 编程生态中的定位，避免与宿主（Kiro、Claude Code、Codex、Cursor）、spec 系统（OpenSpec、Spec Kit）、agent 平台（OpenClaw、Hermes Agent）或方法论生态（Superpowers）竞争。

## 决策

**Sopify 的 durable core 是跨宿主 AI 工作流的证据与授权层。**

它不负责生成代码或编排 agent，而是把外部生产、验证、知识工具的结果收敛成可恢复、可审计、可授权的机器事实。这里的"授权"不是安全/权限授权，而是**判定行动是否可执行、方案是否可归档、交接是否可信**的 authority。

Sopify 官方在 core 之上提供轻量、可插拔、收敛式的 blueprint-driven workflow 作为默认产品体验。

Core 职责：

1. **证据规范**：定义任务/方案/交接/归档事实的标准格式（`.sopify-skills/` 纯文件协议）
2. **授权判定**：Validator 是唯一授权者——判定行动是否可执行、方案是否可归档
3. **收据生成**：fail-closed 授权回执让每次决策可追溯、可审计
4. **跨宿主接力**：handoff 机器契约让任务在不同 session/model/host 间精确恢复
5. **知识沉淀**：跨任务可复用的稳定结论沉淀为 blueprint / history

**外插原则**：谁负责"把事做好"（生产、验证、知识处理），谁外插；谁负责"把结果变成可验证事实"，谁进 core。

不做的事：

| 领域 | 不做 | 归属 |
|------|------|------|
| 模型推理 | LLM 调用、代码生成 | 宿主 |
| 代码执行 | 文件编辑、终端操作 | 宿主 |
| Agent orchestration | 多 agent 编排、skill routing | Agent 平台 |
| 技能市场 | skill 分发、安装、计费 | 独立生态 |
| 方法论 | spec 写作、设计模式教学 | Spec 系统 / 独立产品 |

### 产品栈分层

Core 之上，Sopify 官方提供 **Default Workflow**（blueprint 驱动的收敛式工作流）作为默认产品体验：

- blueprint 驱动分析 → 计划讨论（含跨宿主交叉审查）→ 标准方案包 → 开发验证 → 归档回写
- Default Workflow 消费 Core 契约，不自行定义授权语义
- 地位高于参考 demo，低于 Core

Core + Default Workflow 之外的生产/验证/知识增强，归入 **Plugins / Skills** 层（Sopify 定义接入点，不做 skill 分发）。

## 生存性测试

2027 年宿主原生支持 plan/checkpoint/multi-agent 后，Sopify 仍必须保留：

- 便携式授权证据（跨宿主可验证的 receipt）
- 跨宿主连续工作（不绑定单一 host 的可恢复接力）
- 可审计决策链（独立于宿主的 audit trail）
- 独立验证/合规套件（外部 validator 兼容性）
- 知识飞轮（跨任务可复用的 machine truth 沉淀）

如果以上任一能力被宿主完全替代且无跨宿主可携带性需求，该能力应 sunset。

**Feature intake gate**：每个新特性必须回答——它增强了哪项不可替代能力？是否扩大了 runtime surface？如果只在单宿主场景下有价值，不进 core。

## 后果

- Core（durable）只保留证据规范 / 授权判定 / 收据生成 / 合规验证 / 跨宿主接力契约
- Default Workflow 是 Core 之上的官方收敛式工作流，地位高于参考 demo，低于 Core
- 具有独立用户价值的生产/验证/知识能力设计为外插 Plugins / Skills
- CrossReview 是已实现的外插参考范本
- Runtime 是参考实现，服务于 Default Workflow，不是 co-equal product
- Core promotion rule：只有影响跨宿主互操作、receipt validity、archive admissibility 的契约才进 Core
